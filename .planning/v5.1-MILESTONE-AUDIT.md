## Critical Gaps (Blockers)

### GAP-1: `store.reset()` Crashes Reset Button

**File:** `static/js/main.js:112`
**Severity:** HIGH

`main.js` calls `store.reset()` inside the Reset Session button handler, but `state.js` exposes only `getState`, `subscribe`, `setState`, `setSelections`, `notify` — no `reset` method.

```js
// main.js ~line 108-116
resetBtn.addEventListener('click', () => {
    if (confirm('Reset session? ...')) {
        localStorage.removeItem('jdb_session_id');
        localStorage.removeItem('classification_cache');
        store.reset();           // TypeError: store.reset is not a function
        window.location.reload(); // NEVER REACHED
    }
});
```

**Impact:** User clicks Reset, confirms the dialog, localStorage is cleared, but the page never reloads and the app state is not reset. Session appears to continue normally despite the user's intent to start fresh.

**Fix:** Add `reset()` method to `state.js` store, or replace `store.reset()` with `store.setState(defaultState)` (if `setState` accepts a full state object) or remove the call and rely on `window.location.reload()` alone.

---

### GAP-2: `buildExportRequest()` Drops Three Selection Sections

**File:** `static/js/export.js:buildExportRequest()` (lines ~56-80)
**Severity:** HIGH

`buildExportRequest()` iterates `Object.entries(state.selections)` and for each section does:
```js
const sectionData = profile[sectionId];
if (!sectionData || !sectionData.statements) return;  // silent skip
```

Three section keys do not follow the `profile[sectionId].statements` layout:
- `core_competencies` → lives at `profile.reference_attributes.core_competencies[]` (plain strings)
- `abilities` → lives inside `profile.skills.statements` filtered by `source_attribute === "Abilities"`
- `knowledge` → lives inside `profile.skills.statements` filtered by `source_attribute === "Knowledge"`

`assembleJDPreview()` and `updateSidebar()` both have explicit special-casing for these three sections. `buildExportRequest()` does not.

**Impact:** Any statements the user selected in the Core Competencies, Abilities, or Knowledge tabs are silently absent from the downloaded PDF, DOCX, and JSON. The modal preview correctly shows them, creating a discrepancy between preview and download.

**Fix:** Add special-case handling in `buildExportRequest()` mirroring the logic already in `assembleJDPreview()`:
- `core_competencies`: look up from `profile.reference_attributes.core_competencies`
- `abilities` / `knowledge`: filter `profile.skills.statements` by `source_attribute`
