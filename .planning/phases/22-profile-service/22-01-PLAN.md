**2. Create src/services/profile_parquet_reader.py:**
A new service module that reads the 5 oasis_* parquet files for a given NOC code and returns CoverageStatus-tagged results per tab.

Key implementation details:

**CRITICAL: Use column `oasis_code` (NOT `unit_group_id`, NOT `oasis_profile_code`) for all oasis_* file lookups.** This is confirmed in DATA-INVENTORY.md. Using the wrong column returns NOT_FOUND for all 900 profiles.

Constants:
- `OASIS_CODE_COL = "oasis_code"` -- the lookup column for all 5 oasis_* files
- `OASIS_FILES` dict mapping tab keys to file paths under JOBFORGE_GOLD_PATH:
  - `"skills"` -> `oasis_skills.parquet`
  - `"abilities"` -> `oasis_abilities.parquet`
  - `"knowledge"` -> `oasis_knowledges.parquet` (note: "knowledges" with an 's')
  - `"work_activities"` -> `oasis_workactivities.parquet`
  - `"work_context"` -> `oasis_workcontext.parquet`
- `METADATA_COLS` set of columns to exclude from dimension extraction: `{"unit_group_id", "noc_element_code", "oasis_code", "oasis_label", "_source_file", "_ingested_at", "_batch_id", "_layer"}`

Functions:

`noc_to_oasis_code(noc_code: str) -> list[str]`:
- Takes a 5-digit NOC code (e.g. '21211')
- Returns candidate oasis codes: `["{base}.00", "{base}.01"]`
- Normalize input: `base = noc_code.split('.')[0]` (handles case where code already has decimal)

`get_profile_tab(tab_key: str, oasis_profile_code: str) -> ParquetResult[pd.DataFrame]`:
- Validates tab_key against OASIS_FILES keys; returns LOAD_ERROR for unknown keys
- Calls `lookup_profile(path, OASIS_CODE_COL, oasis_profile_code)` from parquet_reader.py
- Returns the ParquetResult directly (no caching here -- parquet_reader.py handles caching)

`extract_dimension_ratings(df_row: pd.DataFrame, tab_key: str) -> list[dict]`:
- Takes the DataFrame from a FOUND ParquetResult (typically 1 row)
- Iterates over columns, skipping METADATA_COLS
- For each non-metadata column with a non-null, non-zero integer value, creates `{"name": col, "level": int(val)}`
- Sorts by level descending
- Returns empty list if no non-zero values (this signals EMPTY_VALID)
- Do NOT strip whitespace -- read_parquet_safe() already did this

`get_all_profile_tabs(noc_code: str) -> dict[str, tuple[list[dict], str]]`:
- Convenience function called by mapper.py
- For each of the 5 tab keys, tries noc_to_oasis_code candidates in order
- For each candidate: call get_profile_tab, then extract_dimension_ratings
- If ratings are non-empty: return `(ratings, "jobforge")` for that tab
- If FOUND but empty after filtering (EMPTY_VALID): fall through to return `([], "oasis")` -- signals mapper to use OASIS path
- If NOT_FOUND for .00: try .01; if still NOT_FOUND: return `([], "oasis")`
- If LOAD_ERROR: log warning, return `([], "oasis")`
- Returns dict like `{"skills": ([...ratings...], "jobforge"), "abilities": ([], "oasis"), ...}`

Import from parquet_reader.py: `lookup_profile`
Import from config.py: `JOBFORGE_GOLD_PATH`
Import from models.parquet: `CoverageStatus, ParquetResult`
Use stdlib logging: `logger = logging.getLogger(__name__)`

For **skills** -- modify `_map_skills_enriched()`:
- Check `parquet_tabs["skills"]` -- if ratings are non-empty and source is "jobforge":
  - Convert ratings to EnrichedNOCStatement objects (text=name, source_attribute="Skills", source_url=base_url, proficiency with level/max/label/dimension)
  - Set data_source="jobforge"
- Else: use existing OASIS path (current code), data_source="oasis"
- IMPORTANT: The current `_map_skills_enriched` combines Skills + Abilities + Knowledge into one element. Keep this behavior for the `skills` key in the response (backward compat with existing accordion.js filtering). BUT: for the parquet path, the `skills` key should ONLY contain Skills statements (not Abilities/Knowledge).

For **abilities** and **knowledge** -- these are currently served from the combined `skills` element via source_attribute filtering in accordion.js. With parquet, they get their own data:
- Check `parquet_tabs["abilities"]` / `parquet_tabs["knowledge"]`
- If jobforge: build EnrichedJDElementData with own statements list and data_source="jobforge"
- If oasis: filter from OASIS skills data by source_attribute (existing behavior)
- Since abilities and knowledge are NOT top-level keys in ProfileResponse yet, the cleanest approach is: keep the combined `skills` field unchanged for backward compatibility, but modify how the data flows.

**IMPORTANT DECISION on abilities/knowledge separation:**
The current `_map_skills_enriched()` combines all three into `skills`. The accordion.js already filters by `source_attribute`. For Phase 22:
- When parquet data is available: build separate EnrichedNOCStatement lists per tab with correct source_attribute values ("Skills", "Abilities", "Knowledge") -- the existing filter in accordion.js will work
- When OASIS fallback: use existing combined path (already has correct source_attribute per statement)
- Set data_source on the combined `skills` element. When some sub-tabs are parquet and others OASIS, use "jobforge" if at least Skills itself came from parquet (the badge shows per the `skills` element level, and abilities/knowledge filter from it)

For **work_activities** -- modify `_map_key_activities_enriched()`:
- The key_activities element combines Main Duties (unconditional OASIS) + Work Activities
- Check `parquet_tabs["work_activities"]`: if ratings non-empty and source is "jobforge":
  - Build Work Activities statements from parquet ratings
  - Combine with Main Duties from OASIS (Main Duties is ALWAYS OASIS)
  - data_source: set to "oasis" since Main Duties is always OASIS (this element is mixed-source; the badge should reflect the dominant/unconditional OASIS source for Main Duties)
- Else: use existing OASIS path entirely

For **work_context** (effort + responsibility tabs):
- These are already served from parquet via labels_loader.get_work_context_filtered() (existing pre-Phase 22 code!)
- Check `parquet_tabs["work_context"]`: if source is "jobforge", set data_source="jobforge" on the effort and responsibility EnrichedJDElementData objects
- If LOAD_ERROR, fall back to OASIS-scraped work context
- The mapper._map_effort_enriched() and _map_responsibility_enriched() already use labels_loader which reads oasis_workcontext.parquet. These continue to work as-is. The only change is adding data_source="jobforge" (since labels_loader reads from parquet, this IS jobforge data)
- NOTE: labels_loader reads parquet via its own pd.read_parquet() cache, while profile_parquet_reader uses parquet_reader._cache. This creates two separate caches for the same file. This is acceptable per research recommendation (file is small, avoids labels_loader refactor).
