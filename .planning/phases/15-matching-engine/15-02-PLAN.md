class SemanticMatcher:
    """Calculate semantic similarity between JD text and occupational group definitions."""

    def __init__(self):
        # Use all-MiniLM-L6-v2: fast, good accuracy, no GPU needed (per RESEARCH.md)
        self.model = SentenceTransformer('all-MiniLM-L6-v2')
        self._definition_cache = {}  # Cache embeddings for group definitions

    def compute_similarity(self, jd_text: str, definition: str) -> float:
        """Calculate cosine similarity between JD and definition."""

    def shortlist_by_definition(
        self,
        jd_text: str,
        groups: List[Dict],
        min_similarity: float = 0.3
    ) -> List[Dict]:
        """
        Return groups with semantic similarity >= threshold.

        Args:
            jd_text: Combined Client-Service Results + Key Activities text
            groups: List of group dicts with 'definition' key
            min_similarity: Threshold (default 0.3 per CONTEXT.md)

        Returns:
            List of dicts with: group, semantic_similarity, ranked by similarity desc
        """
```

**shortlist_candidates() function:**
Convenience function that:
1. Takes jd_text and groups list
2. Uses SemanticMatcher to compute similarities to definitions
3. Returns top 10 candidates sorted by semantic_similarity

Per CONTEXT.md decision: "Inclusion statements: Use for shortlisting candidates only, not for scoring or evidence"

This means:
- Inclusions CAN be used to identify viable candidate groups (part of shortlist filtering logic)
- Inclusions should NOT contribute to combined_score used for ranking
- The combined_score is based on definition similarity only

Scoring formula:
- combined_score = semantic_similarity (100% definition-based)

Inclusions are used in a separate filter step: if a group's inclusions match the JD text,
it passes the shortlist filter even if definition similarity is borderline. But the
ranking score remains definition-only.

Cache definition embeddings since groups don't change often. JD embeddings computed fresh each call.

        Build mapping from group codes to associated labels.

        NOTE: labels_loader uses OaSIS profile codes, not TBS group codes.
        We need a mapping strategy:

        Option A: If labels contain group code references, extract them
        Option B: Use semantic similarity between labels and group definitions
        Option C: Return empty mapping if no clear connection exists

        For v4.0, use Option C - return 0 boost if we can't establish mapping.
        Future: Add explicit TBS group -> OaSIS code mapping table.
        """

    def get_boost(self, group_code: str, jd_text: str) -> float:

        Per CONTEXT.md: "boost confidence if labels.csv supports the match"
        Max boost: 0.1 (10%)

        Returns 0.0 if:
        - No labels mapping established
        - Group not in mapping
        - Labels don't semantically match JD text
        """

def shortlist_with_all_signals(
    jd_text: str,
    groups: List[Dict],
    min_similarity: float = 0.3,
    max_candidates: int = 10
) -> List[Dict]:
    """
    Shortlist candidates using all available signals.

    Per CONTEXT.md decision: "Inclusion statements: Use for shortlisting candidates only,
    not for scoring or evidence"

    Signal usage:
    1. Semantic similarity to DEFINITION - primary ranking signal
    2. Semantic similarity to INCLUSIONS - used for FILTERING only, not scoring
       (a group with inclusion match passes shortlist even if definition similarity is borderline)
    3. Labels.csv boost - additive boost for ordering

    Args:
        jd_text: Combined Client-Service Results + Key Activities
        groups: List of group dicts from repository (with inclusions)
        min_similarity: Threshold (default 0.3 per CONTEXT.md)
        max_candidates: Max candidates to return (default 10)

    Returns:
        List of dicts sorted by combined_score desc:
        {
            'group': original group dict,
            'semantic_similarity': float,  # similarity to definition
            'inclusion_match': bool,       # True if inclusions helped shortlist
            'labels_boost': float,
            'combined_score': float        # definition_similarity + labels_boost ONLY
        }
    """
```

Scoring formula (per CONTEXT.md):
- combined_score = semantic_similarity + labels_boost

IMPORTANT: Inclusions do NOT contribute to combined_score.
Inclusions are used for filter logic only:
- If definition_similarity >= min_similarity, include the candidate
- If definition_similarity < min_similarity BUT inclusion_match is True,
  ALSO include the candidate (inclusion helped shortlist)

This respects the user decision: inclusions identify candidates (shortlisting role)
but don't boost their ranking score.

Filter to candidates with combined_score >= min_similarity, return top max_candidates.
